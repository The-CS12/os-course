توی فصل قبل با نحوه جا به جایی بین پراسس ها آشنا شدید!‌ ولی الان که میدونید ترد ها چی هستن باید یه سوال براتون ایجاد شده باشه:
> **پس Context Switch روی Thread ها انجام میشه؟؟🤔** <br>

بله دقیقا❗ <br>
وقتی پردازنده  Context Switch انجام میده عملا داره بین Thread ها سویچ میکنه. Process ها خودشون اجرا نمیشن بلکه ترد های درون اون ها اجرا میشن. برای همینه که به ترد ها **Execution Unit** یا واحد اجرایی میگن. <br>

{{< readmore type="" title="یکم بیشتر بدونید" open="false" >}}ما دو Context Switch داریم: 
- **Process Context Switch**
- **Thread Context Switch**

(البته اینجا اشتباه نشه به صورت فنی و عملی پردازنده در هر دو حالت بین Thread ها سویچ میشه❗)
{{< /readmore >}}



پردازنده هیچوقت نمیگه **"میخوام Process1 رو اجرا کنم!"** <br>
بلکه میگه **"میخوام ترد شماره فلان رو از Process1 اجرا کنم!"**
#### Process Context Switch   ⚡VS⚡   Thread Context Switch
وقتی پردازنده میخواد Context Switch انجام بده، اگر تردی که میخواد بهش سویج کنه جزو ترد های پراسس در حال اجرا (پراسسی که تردش داره توسط CPU اجرا میشه) باشه **Thread Context Switch** اتفاق افتاده. یعنی تردی از همون پراسس رو داره اجرا میکنه! <br>
ولی اگر پردازنده بخواد Context Switch انجام بده و تردی که قراره بهش سویچ کنه جزو پراسس در حال اجرا (پراسسی که تردش داره اجرا میشه) نباشه **Process Context Switch** اتفاق افتاده. یعنی پردازنده باید به تردی از یه پراسس دیگه سویچ کنه. این عمل نسبت به Thread Context Switch سنگین تره!

{{< readmore type="info" title="چرا Prcoess Context Switch از Thread Context Switch سنگین تره؟" open="false" >}}
وقتی پردازنده بین ترد های یه پراسس سویچ کنه, از اونجایی که پراسس هاشون یکیه, اطلاعات پراسس, کد و فضای آدرس یکسانی دارن پس نیازی به تغییر این اطلاعات نیست ولی اگر پردازنده بخواد تردی از پراسس دیگه ای رو اجرا کنه باید تمامی اطلاعات Process (Process Image) رو لود, فضای آدرس و رجیستر های مربوط رو Owerwrite کنه و این عمل هم سرعت کمتری داره هم سنگین تره❗
